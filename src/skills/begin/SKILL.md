# Begin — 작업 시작 문서 생성

유의미한 작업을 시작할 때 4개의 추적 문서를 자동 생성한다.
계획 → 실행 → 검증 사이클에서 컨텍스트 유실을 방지한다.

## 명령어

| 명령 | 설명 | 예시 |
|------|------|------|
| `"작업명"` | 작업 디렉토리 + 4개 문서 생성 | `/begin "스킬셋 점검"` |
| `status` | 현재 활성 작업 목록 표시 | `/begin status` |
| `close "작업명"` | 작업 완료 처리 (진행문서에 기록) | `/begin close "스킬셋 점검"` |

## 생성되는 문서

```
.claude/tasks/<작업명-kebab>/
├── plan.md        — 목표, 범위, 우선순위, 단계별 계획
├── progress.md    — 현재 상태, 완료 항목, 다음 단계
├── context.md     — 결정 사항, 참조 정보, 발견한 것
└── failures.md    — 시도한 것, 실패 원인, 교훈
```

## 실행 로직

### `__ARGS__` 파싱

```
RAW_ARGS="__ARGS__"

case:
  "status"          → 활성 작업 목록
  "close ..."       → 작업 완료 처리
  "" (빈값)         → 활성 작업 목록 (status와 동일)
  그 외             → 작업 생성 플로우
```

### `"작업명"` — 작업 생성

#### Step 1: 작업명 → 디렉토리명 변환

작업명을 kebab-case로 변환하여 디렉토리를 생성한다.
- 한글은 그대로 유지 (폴더명에 한글 허용)
- 공백 → 하이픈, 특수문자 제거
- `--scope` 옵션이 있으면 plan.md에 반영

```bash
TASK_DIR=".claude/tasks/<kebab-변환된-작업명>"
mkdir -p "$TASK_DIR"
```

이미 존재하면 "이미 진행 중인 작업입니다. 문서를 열까요?" 확인.

#### Step 2: 4개 문서 생성

**plan.md**:
```markdown
# <작업명> — 계획

## 목표
(작업 시작 시 채운다)

## 범위
(--scope 옵션이 있으면 여기에 반영)

## 우선순위별 작업

### P0 — 즉시
- [ ]

### P1 — 중요
- [ ]

### P2 — 개선
- [ ]

## 제약
- 한 커밋 = 한 목적
- 정리와 기능을 한 커밋에 섞지 않기
```

**progress.md**:
```markdown
# <작업명> — 진행

## 현재 상태: 시작 전

## 완료된 작업
(없음)

## 진행 중
(없음)

## 다음 단계
- plan.md 작성
```

**context.md**:
```markdown
# <작업명> — 컨텍스트

## 핵심 결정 사항
(작업 중 내린 결정을 기록)

## 참조
(관련 문서, URL, 커밋 해시 등)

## 발견한 것
(작업 중 알게 된 사실)
```

**failures.md**:
```markdown
# <작업명> — 시도 및 실패 일지

> 시도한 것, 실패한 것, 원인, 교훈을 기록한다.

## 기록

(아직 없음)

<!--
기록 형식:
## [날짜] 시도 제목
- **시도**: 무엇을 했는가
- **결과**: 성공/실패
- **원인**: 왜 실패했는가
- **교훈**: 다음에 어떻게 할 것인가
-->
```

#### Step 3: 완료 보고

```
작업 환경 생성 완료:
- 디렉토리: .claude/tasks/<작업명>/
- plan.md: ✓ (목표·범위 작성 필요)
- progress.md: ✓
- context.md: ✓
- failures.md: ✓

다음: plan.md에 목표와 범위를 작성하세요.
```

### `status` — 활성 작업 목록

```bash
ls -d .claude/tasks/*/
```

각 작업의 `progress.md`에서 "현재 상태:" 줄을 읽어 요약 표시.

### `close "작업명"` — 작업 완료

1. 해당 작업의 `progress.md`에 "현재 상태: 완료"로 업데이트
2. 완료 일시 기록
3. 디렉토리는 삭제하지 않음 (참조용 보존)

## 문서 활용 규칙

작업 진행 중 다음 시점에 문서를 업데이트한다:

| 시점 | 업데이트 대상 |
|------|-------------|
| 작업 방향 결정 시 | plan.md (체크박스 체크) |
| 단계 완료/시작 시 | progress.md |
| 새로운 사실 발견 시 | context.md |
| 시도가 실패했을 때 | failures.md |
| 커밋할 때 | progress.md (완료 항목 이동) |

## Gotchas

> Claude가 작업 문서에서 자주 실수하는 것. 실패할 때마다 한 줄 추가.

1. **문서 생성만 하고 내용을 안 채움** — plan.md의 목표·범위는 생성 직후 사용자와 함께 작성해야 한다.
2. **progress.md 업데이트 누락** — 단계 완료 시 반드시 progress.md를 업데이트. 머릿속으로만 추적하지 않기.
3. **failures.md를 안 씀** — 실패는 가장 중요한 학습 자료. 시도-실패-원인-교훈 형식으로 반드시 기록.
4. **작업명을 너무 길게** — kebab-case로 3-4단어 이내. "스킬셋-점검" ✓, "글렌-에이전트-설정-프로젝트의-스킬-에이전트-커맨드-전수-점검" ✗
