# TDD Workflow

Red → Green → Refactor 사이클을 강제한다. **일반적인 TDD 교과서 내용은 적지 않는다.**
Claude가 구체적으로 틀리는 부분만 교정한다.

## 사이클

1. **Red**: 실패하는 테스트를 먼저 작성 (구현 코드 없이)
2. **Green**: 테스트를 통과시키는 최소한의 코드 작성
3. **Refactor**: 테스트 green 유지하며 정리

## 핵심 규칙

- 커버리지 80%+ 필수 (`npm run test:coverage`)
- 버그 수정은 **재현 테스트 먼저** → 그 다음 수정
- E2E는 critical user flow만 (회원가입, 결제, 핵심 비즈니스 흐름)
- 테스트 파일은 소스 옆에 배치: `Button.tsx` → `Button.test.tsx`

## Gotchas

> Claude가 TDD에서 자주 실수하는 것. 실패할 때마다 한 줄 추가.

1. **테스트를 코드 다음에 작성** — Claude는 습관적으로 구현부터 짜고 "이제 테스트를 추가하겠습니다"라고 한다. 반드시 테스트 파일부터 열어야 한다.
2. **과도한 모킹** — 외부 서비스만 모킹. 내부 모듈까지 모킹하면 리팩토링 시 테스트가 의미 없어진다.
3. **구현 세부사항 테스트** — `component.state.count === 5` 대신 `screen.getByText('Count: 5')` 등 사용자 관점으로 테스트.
4. **테스트 간 의존성** — 각 테스트가 독립적이어야 한다. `beforeEach`로 상태 초기화 필수.
5. **비동기 타이밍 하드코딩** — `waitForTimeout(3000)` 대신 `waitFor(() => expect(...))` 사용.
6. **한 테스트에 여러 assert** — 하나의 행동을 검증하는 관련 assert는 OK, 서로 다른 행동을 한 테스트에 넣지 않기.
7. **스냅샷 남용** — 스냅샷 테스트는 의도적 변경과 실수를 구분 못한다. 행동 기반 assert 우선.
