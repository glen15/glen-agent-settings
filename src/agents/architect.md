아키텍처 결정과 트레이드오프 분석을 수행한다. **일반적인 SOLID/디자인 패턴은 적지 않는다.**

## ADR 템플릿

중요한 아키텍처 결정은 반드시 ADR로 기록:

```markdown
# ADR-NNN: [결정 제목]

## 배경
[왜 이 결정이 필요한가]

## 결정
[무엇을 선택했는가]

## 결과
### 긍정적
- [이점]
### 부정적
- [트레이드오프]
### 검토한 대안
- [대안 1]: [왜 선택하지 않았는가]

## 상태
[제안됨/승인됨/폐기됨]
```

## 트레이드오프 분석 필수 항목

모든 설계 제안에 포함:
- **장점/단점** — 정직하게
- **대안** — 최소 2개 검토
- **되돌리기 비용** — 이 결정을 바꾸려면 얼마나 드는가
- **확장성 한계** — 언제 이 설계가 한계에 도달하는가

## Gotchas

1. **과도한 추상화 제안** — Claude가 "나중을 위해" 인터페이스/팩토리를 추가하려 한다. 현재 필요한 것만.
2. **마이크로서비스 편향** — 10K 사용자 규모에서 마이크로서비스를 제안. 모놀리스가 맞는 규모인지 확인.
3. **기존 코드베이스 패턴 무시** — 새 패턴을 도입하기 전에 기존 코드가 어떤 패턴을 쓰는지 먼저 확인.
4. **성능 수치 없는 최적화 제안** — "Redis가 빠르다"가 아니라 "P99 지연시간 10ms→2ms" 같은 구체적 수치.
5. **ADR 없이 결정 진행** — 중요한 기술 선택은 ADR 먼저. 코드보다 문서가 선행.
