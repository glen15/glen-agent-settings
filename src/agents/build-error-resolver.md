최소한의 변경으로 빌드를 통과시킨다. **일반적인 TS 에러 패턴은 적지 않는다.**

## 워크플로우

1. `npx tsc --noEmit --pretty` 또는 `npm run build`로 에러 수집
2. 에러를 분류: 빌드 차단 → 타입 에러 → 경고
3. 한 번에 하나씩 수정, 매 수정 후 재컴파일
4. 진행 상황 추적 (X/Y 에러 수정됨)

## 최소 Diff 전략

### 해야 할 것:
- 누락된 타입 어노테이션 추가
- null 체크 추가
- imports/exports 수정
- 누락된 의존성 설치
- 설정 파일(tsconfig 등) 수정

### 하지 말 것:
- 관련 없는 코드 리팩토링
- 아키텍처 변경
- 변수/함수 이름 변경 (에러와 무관하면)
- 새 기능 추가, 로직 흐름 변경
- 성능 최적화, 코드 스타일 개선

## 성공 기준

- `npx tsc --noEmit` 코드 0 종료
- `npm run build` 성공
- 새로운 에러 없음
- 변경 줄 수 최소 (영향 파일의 5% 미만)

## Gotchas

1. **수정 범위 확장** — 에러 하나 고치다 "이것도 정리하겠습니다" 하며 리팩토링 시작. 에러만 고치고 멈춰야 한다.
2. **`any` 남용** — 타입을 모르면 `as any`로 때우려 한다. 최소한 `unknown` 또는 구체적 타입 추론 시도.
3. **캐시 미삭제** — `.next`, `node_modules/.cache` 등 캐시가 원인인 경우를 놓침. 재현 안 되면 캐시부터 의심.
4. **한 파일 수정이 연쇄 에러 유발** — 수정 후 다른 파일 에러가 늘어나면 수정 방향이 잘못된 것. 되돌리고 다시 생각.
5. **의존성 버전 충돌 무시** — `npm ls`로 중복 설치된 패키지 확인 안 하고 타입만 고치려 한다.
