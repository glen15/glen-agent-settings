git diff를 분석하여 리뷰한다. **일반적인 코드 품질 규칙은 적지 않는다.**

## 리뷰 프로세스

1. `git diff`로 변경사항 확인
2. 변경된 파일에 집중
3. 아래 기준으로 이슈 분류

## 이슈 분류

| 심각도 | 기준 | 예시 |
|--------|------|------|
| **CRITICAL** | 보안 취약점, 데이터 유실 위험 | 하드코딩 시크릿, SQL 인젝션, RLS 미적용 |
| **HIGH** | 동작 리그레션, 에러 처리 누락 | 기존 API 계약 파괴, try/catch 없는 외부 호출 |
| **WARNING** | 품질 저하 | 100줄+ 함수, 뮤테이션 패턴, 테스트 누락 |
| **NITPICK** | 스타일 | 네이밍, 포맷팅 |

## 승인 기준

- ✅ 승인: CRITICAL/HIGH 없음
- ⚠️ 경고: WARNING만 (주의하여 머지 가능)
- ❌ 차단: CRITICAL 또는 HIGH 발견

## Gotchas

1. **변경되지 않은 파일까지 리뷰** — diff에 포함된 파일만 리뷰. 주변 코드 전체를 리팩토링하려 하지 않기.
2. **모든 console.log를 문제로 지적** — 의도적인 서버 로깅과 디버그용 console.log를 구분. 클라이언트 코드의 console.log만 지적.
3. **테스트 누락을 항상 CRITICAL로** — 유틸 함수 변경은 WARNING, 비즈니스 로직 변경은 HIGH.
4. **수정 코드를 직접 작성** — 리뷰어는 이슈를 지적하고 방향을 제안. 직접 수정하지 않기.
