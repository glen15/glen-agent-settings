# Anthropic의 1M 컨텍스트 윈도우가 Claude Code 경제학과 제어를 어떻게 바꾸는가

**2026년 3월 13일부터 정식 출시된 1M 컨텍스트 윈도우는 Claude Code 루프의 토큰 경제학을 근본적으로 변화시킨다.** 매 API 호출이 전체 대화 히스토리를 재전송하므로, 500K 토큰에서 실행되는 세션은 100K일 때보다 요청당 약 5배 더 많은 토큰을 소비한다—이것이 구독 리밋 소진을 직접 가속시킨다. Claude Code에는 `--max-context` 플래그가 없지만, `--output-format json` 토큰 리포팅, `--max-turns`, `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE`의 조합이 실용적인 프로그래밍적 제어를 제공한다. 결정적으로, `claude -p` 새 컨텍스트 루프(`--resume` 없이)는 컨텍스트 성장을 완전히 회피한다: 매 호출이 깨끗하게 시작되어 자율 워크플로우에서 가장 토큰 효율적인 패턴이다.

---

## 우려 1: 1M 컨텍스트가 구독 할당량을 더 빨리 소진시키는가 — 그렇다, 수학이 증명한다

메커니즘은 직관적이며 Anthropic의 문서와 사용자 보고 모두에서 확인된다. **매 Claude Code API 호출이 전체 누적 컨텍스트를 입력 토큰으로 전송한다.** 400K 토큰의 컨텍스트에 있는 세션은 100K일 때보다 요청당 4배 더 많은 입력 토큰을 보낸다. Claude Max 구독 리밋이 토큰 기반이므로(메시지 기반이 아님), 더 큰 컨텍스트가 할당량을 비례적으로 빠르게 소모한다.

Anthropic의 2026년 3월 13일 업데이트는 **Opus 4.6과 Sonnet 4.6에 표준 가격으로** 1M 컨텍스트를 정식 출시했다—각각 MTok당 $5/$25, $3/$15이며, 어떤 길이에서도 롱 컨텍스트 프리미엄이 없다. 900K 토큰 요청이 9K 토큰 요청과 동일한 토큰당 요율을 적용받는다. 구형 모델(Sonnet 4.5, Sonnet 4)은 200K 토큰 이상에서 여전히 **입력 2배 / 출력 1.5배 프리미엄**을 부과하며 `context-1m-2025-08-07` 베타 헤더가 필요하다.

실제 증거는 극명하다. GitHub Issue #28723의 Max 5x 구독자가 `claude-opus-4-6[1m]`을 활성화하고 컨텍스트가 약 320K 토큰(약 32% 채움)인 상태에서 **세션 할당량이 약 50분 만에 8%에서 67%로 점프**하는 것을 목격했다. Reddit 개발자들은 Opus로 아키텍처 리팩토링 세션에서 "3개 프롬프트에 4시간 사용량"을 소모했다고 보고한다. 8개월간 100억 토큰을 추적한 한 개발자는 API 등가 비용이 $15,000이었을 것이라고 발견했다—Max에서 $800을 지불했지만, Sonnet 중심 사용이 요청당 컨텍스트를 작게 유지했기 때문이었다.

**프롬프트 캐싱이 의미 있는 완화를 제공한다.** 캐시된 토큰은 기본 입력 가격의 10%만 비용이 들고, Claude Code는 시스템 프롬프트 같은 반복 콘텐츠를 자동 캐싱한다. 하지만 매 턴마다 커지는 대화 히스토리는 완전히 캐싱할 수 없다. 순 효과: 1M 컨텍스트가 압축 관련 정보 손실을 제거하지만, 그 대가로 상당히 빠른 할당량 소비를 가져온다. 기존 경제학을 원하는 사용자는 `CLAUDE_CODE_DISABLE_1M_CONTEXT=1`을 설정할 수 있다.

## 우려 2: 성능 저하 — 25% 미만이라 해도 밤샘 루프에서는 중요하다

Anthropic은 1M 컨텍스트에서 성능 저하가 25% 미만이라고 주장한다 (경쟁사 모델은 50% 이상). 이것은 일반 사용에서는 인상적이지만, 밤샘 자율 루프에서는 다른 계산이 된다.

100회 반복 루프에서 각 반복마다 5%의 미세한 성능 저하가 있다면, 누적 효과는 무시할 수 없다. 에이전트가 아키텍처 결정을 내리는 반복에서 5% 더 많은 환각이 생기면, 그 위에 쌓이는 후속 코드가 전부 오염된다. **밤샘 루프에서 성능 저하의 영향은 곱셈적이지, 덧셈적이 아니다.**

이것이 바로 `claude -p` 새 세션 루프가 결정적인 이유다: 매 반복이 컨텍스트 0에서 시작하므로, 1M 윈도우의 성능 저하 문제를 아예 겪지 않는다.

## 핵심 질문: 컨텍스트에 임계값을 걸 수 있는가?

### 직접적인 `--max-context` 플래그는 없다

Claude Code에는 컨텍스트 크기를 직접 제한하는 플래그가 존재하지 않는다. 하지만 여러 메커니즘을 조합하면 효과적인 제어가 가능하다.

### 방법 1: `--output-format json`으로 토큰 파싱 후 루프 종료 (가장 권장)

`--output-format json`은 매 호출의 결과에 `usage` 객체를 포함시킨다: `input_tokens`, `cache_creation_input_tokens`, `cache_read_input_tokens`, `output_tokens`, 그리고 `total_cost_usd`와 `session_id`. `--resume`으로 세션 연속성을 유지하는 bash 루프에서 이 필드를 `jq`로 파싱하여 대략적 컨텍스트 크기를 계산하고, 임계값 초과 시 세션을 강제 재시작할 수 있다.

**300K 임계값 구현 예시:**

```bash
#!/bin/bash
THRESHOLD=200000  # 200K 토큰에서 세션 리셋
SESSION_ID=""
ITERATION=0
MAX_ITERATIONS=100
MAX_TURN=20

while [ $ITERATION -lt $MAX_ITERATIONS ]; do
  ITERATION=$((ITERATION + 1))
  echo "[$(date)] 반복 #$ITERATION 시작" >> loop.log

  # 세션 ID가 있으면 resume, 없으면 새 세션
  if [ -n "$SESSION_ID" ]; then
    RESULT=$(cat PROMPT.md | claude -p \
      --output-format json \
      --max-turns MAX_TURN \
      --resume "$SESSION_ID" 2>&1)
  else
    RESULT=$(cat PROMPT.md | claude -p \
      --output-format json \
      --max-turns MAX_TURN 2>&1)
  fi

  # 토큰 사용량 파싱
  INPUT_TOKENS=$(echo "$RESULT" | jq '
    (.usage.cache_read_input_tokens // 0) +
    (.usage.cache_creation_input_tokens // 0) +
    (.usage.input_tokens // 0)')

  echo "[$(date)] 입력 토큰: $INPUT_TOKENS" >> loop.log

  # 임계값 초과 시 세션 리셋
  if [ "$INPUT_TOKENS" -gt "$THRESHOLD" ]; then
    echo "[$(date)] 컨텍스트 ${INPUT_TOKENS} > ${THRESHOLD}. 세션 리셋." >> loop.log
    SESSION_ID=""  # 다음 반복에서 새 세션 시작
  else
    SESSION_ID=$(echo "$RESULT" | jq -r '.session_id')
  fi

  # 완료 확인
  if grep -q '"status": "all_passing"' prd.json 2>/dev/null; then
    echo "[$(date)] 모든 기능 통과 - 루프 완료" >> loop.log
    break
  fi

  sleep 5
done
```

### 방법 2: `--max-turns`로 간접 제어

`--max-turns N`은 반복당 에이전트 사이클을 제한한다. 턴당 평균 토큰 소비를 알고 있다면 (일반적으로 10,000~50,000 토큰/턴), 턴 수를 제한해 간접적으로 컨텍스트 성장을 통제할 수 있다. 초과 시 JSON 출력에 `"subtype": "error_max_turns"`가 포함된다.

### 방법 3: `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE`로 조기 압축

이 환경 변수는 자동 압축이 트리거되는 컨텍스트 윈도우 퍼센트를 제어한다. 1~100 값을 받는다. 기본값은 CLI에서 약 **83.5%**이다.

```bash
# 70%에서 조기 압축 트리거
export CLAUDE_AUTOCOMPACT_PCT_OVERRIDE=70
```

**중요한 제약**: 소스 코드의 `Math.min` 클램프 때문에 기본값(~83.5%) 이상으로 올리는 것은 **효과가 없다**. 이 오버라이드는 임계값을 낮출 수만 있고, 높일 수는 없다.

### 방법 4 (가장 단순하고 효과적): `--resume` 없이 `claude -p` 루프

**이것이 핵심이다.** `--resume` 없이 `claude -p`를 호출하면, 매 반복이 완전히 새로운 컨텍스트에서 시작한다. 컨텍스트가 절대 성장하지 않는다. 반복당 컨텍스트는 시스템 프롬프트(~10~20K 토큰) + CLAUDE.md 파일(~300~2K 토큰) + 파이프된 프롬프트 + 해당 실행 중 도구 호출로만 구성된다.

```bash
# 가장 단순하고 효과적인 패턴
# 컨텍스트 임계값이 필요 없다 — 매 반복이 0에서 시작
while :; do
  cat PROMPT.md | claude -p --max-turns 30 --output-format json \
    >> "logs/iteration-$(date +%s).json" 2>&1

  # 완료 확인
  if grep -q '"status": "all_passing"' prd.json 2>/dev/null; then
    break
  fi

  sleep 5
done
```

이 패턴에서는:
- 컨텍스트 임계값 자체가 불필요하다
- 1M 윈도우의 성능 저하를 겪지 않는다
- 리밋 소비가 예측 가능하고 일정하다
- 상태는 파일(prd.json, progress.txt, git)을 통해 전달된다

## 컨텍스트 성장 메커니즘: 두 패턴의 극명한 차이

### `claude -p` (resume 없음): 컨텍스트가 절대 성장하지 않는다

반복 간 대화 히스토리가 전달되지 않는다. 반복당 컨텍스트는 고정 구성 요소(시스템 프롬프트, CLAUDE.md)와 해당 실행의 도구 호출로만 구성된다. 자동 압축이 트리거될 일이 거의 없다—단일 호출이 윈도우를 채울 가능성이 낮기 때문이다.

### 인터랙티브 세션 또는 `--resume` 체인: 선형적으로 성장

매 사용자 메시지, 어시스턴트 응답, 도구 결과가 전량 누적된다. 도구 출력—파일 읽기, grep 결과, bash 출력—이 Cognition의 분석에 따르면 **컨텍스트의 60~80%를 차지**한다. 단일 "이 파일을 수정해" 명령이 시스템 프롬프트, 히스토리, 파일 읽기, 도구 출력을 포함해 50,000~150,000 토큰을 소비할 수 있다. 15개의 반복 명령 후, 마지막 명령이 **200K+ 입력 토큰**을 보낼 수 있다.

### 자동 압축 트리거 시점

자동 압축은 CLI에서 컨텍스트 윈도우의 약 **83.5%**에서 트리거된다 (VS Code 확장에서는 약 65%). 디컴파일된 소스 코드가 실효 윈도우 크기에서 빼는 13K 토큰 버퍼 상수를 보여준다. `/context`가 보고하는 실용적 버퍼는 약 33K 토큰이다.

| 컨텍스트 윈도우 | 압축 트리거 시점 (약 83.5%) | 실효 작업 공간 |
|---|---|---|
| 200K (기존) | ~167K 토큰 | ~154K 토큰 |
| 1M (신규) | ~835K 토큰 | ~822K 토큰 |

**1M 윈도우에서 압축이 ~835K까지 지연된다.** Anthropic은 **압축 이벤트가 15% 감소**했다고 보고하며, 50~80턴의 일반 세션에서는 압축을 아예 겪지 않을 수 있다.

압축의 3단계 프로세스: 도구 출력이 먼저 삭제되고(가장 큰 소비자), 대화 히스토리가 5~10K 토큰 구조화 요약으로 압축되고, CLAUDE.md 파일이 디스크에서 새로 로드된다. **일반적 감축률은 60~80%**. 살아남는 것: 고수준 작업 요약과 CLAUDE.md 지시. 사라지는 것: 구체적 줄 번호, 정확한 에러 메시지, 디버깅 가설, 아키텍처 결정 근거, 테스트 결과 상세.

## 1M 윈도우가 만드는 양날의 검

1M 컨텍스트에서 압축 임계값이 ~167K에서 ~835K로 이동한다. 이것은 압축으로 인한 정보 손실을 크게 줄이지만, 더 많은 토큰이 정리 전에 누적되어 후속 요청마다 더 큰 입력 페이로드를 전송한다.

하지만 한 분석에서 컨텍스트 윈도우를 200K에서 500K로 올리면 실제로 **총 토큰 소비가 감소**했다는 발견이 있다—에이전트가 압축 후 파일 재읽기에 쓰는 시간이 줄었기 때문이다. "압축 피드백 루프"—압축 후 파일 재읽기가 컨텍스트를 다시 채우고, 또 다른 압축을 트리거하고, 요약의 요약을 만드는—는 1M 컨텍스트가 대체로 제거하는 상당한 낭비 원천이다.

## 실용적 권장 사항 요약

| 시나리오 | 권장 접근법 |
|---|---|
| **밤샘 자율 루프** | `claude -p` (resume 없음) + 파일 기반 상태. 컨텍스트 임계값 불필요 |
| **세션 연속성이 필요한 복잡한 작업** | `claude -p --resume` + JSON 토큰 파싱 + 300K 임계값에서 세션 리셋 |
| **1M 컨텍스트의 리밋 영향 완화** | `CLAUDE_CODE_DISABLE_1M_CONTEXT=1`로 200K로 제한 |
| **단일 세션에서 조기 정리** | `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE=70`으로 70%에서 압축 |
| **병렬화 가능한 작업** | Task 도구(서브에이전트)로 각각 독립 컨텍스트 윈도우 할당 |

핵심을 한 줄로: **`claude -p` 새 세션 루프에서는 1M 컨텍스트 윈도우가 성능 저하도, 리밋 가속도, 압축 문제도 일으키지 않는다—컨텍스트가 반복 간에 성장하지 않기 때문이다. 이것이 밤샘 루프에서 가장 중요한 설계 결정이다.**
