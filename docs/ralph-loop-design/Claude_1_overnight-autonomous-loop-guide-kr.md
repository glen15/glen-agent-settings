# 밤샘 자율 AI 코딩 루프 설계: 완전한 운영 가이드

**장시간 자율 AI 코딩 에이전트는 밤새 안정적으로 실행될 수 있다—단, 세션 회전, 실패 복구, 품질 게이트를 의도적으로 설계한 경우에만.** 지배적인 패턴인 "Ralph 루프"는 매 반복마다 에이전트의 컨텍스트 윈도우를 리셋하면서 git 커밋과 진행 파일을 통해 상태를 영속시킨다. Claude Code, Cursor, Codex를 사용하는 팀들이 7~52시간 세션에서 회당 30K~151K 줄의 코드를 생성한 사례가 문서화되어 있다. 핵심 통찰: 밤샘 성공은 하나의 세션을 계속 살리는 것이 아니라, 짧고 검증 가능한 세션들의 루프를 설계해 파일시스템을 통해 상태를 인수인계하는 데 달려 있다. 이 보고서는 2025~2026년의 실제 배포 사례를 바탕으로 실패 분석, 안정적 아키텍처, 권한, 문서화, 테스트 전략, 운영 절차를 다룬다.

---

## 1) 실패 원인 분석: 밤샘 코딩 세션이 실패하는 구조적 원인

자율 코딩 에이전트는 예측 가능하고 분류 가능한 방식으로 실패한다. 이 실패 모드를 이해하는 것이 안정적인 밤샘 루프 설계의 기초다.

### 왜 중요한가

실패 원인을 모르면 설계가 아니라 운에 의존하게 된다. 구조적 원인을 알면 예방 설계가 가능하다.

### 컨텍스트 윈도우 소진과 드리프트

모든 도구 호출, 에러 트레이스백, 파일 읽기가 컨텍스트 윈도우에 토큰을 추가한다. Claude Code의 실효 작업 메모리는 시스템 프롬프트 오버헤드 이후 약 **176K 토큰**이며, MCP 서버가 로드되면 약 **120K**까지 떨어진다. 용량의 **75~92%**에 도달하면 자동 압축(auto-compaction)이 발동되어 **60~80% 토큰 감축**을 달성하는 요약을 생성하지만, 에이전트가 작업 중간에 핵심 구현 컨텍스트를 잃을 수 있다. SWE-bench Pro에서 **컨텍스트 오버플로가 Claude Sonnet 4 실패의 35.6%**를 차지했다.

더 미묘한 문제는 *지시문 원심분리(instruction centrifugation)*다. 실행 로그가 쌓이면 softmax 어텐션이 최근 토큰에 가중치를 분배해, 원래의 시스템 프롬프트가 기술적으로는 존재하지만 기능적으로는 보이지 않게 된다. 드리프트는 **20~100턴**에서 측정 가능해지고 이후 가속된다. 연구에 따르면 짧은 벤치마크에서 70% 이상 점수를 받는 프론티어 모델이 긴 작업에서는 약 **23%까지 떨어진다**.

### 순환 에러 루프와 환각 연쇄

가장 비용이 큰 실패 모드는 수정-파손 순환이다: 에이전트가 수리를 시도하고, 새 에러를 도입하고, 그것을 다시 고치려 한다. 한 연구에서 분석된 루프의 **45%가 "문제적"으로 분류**되었다. GitHub Copilot Agent 사용자들은 이 순환에서 전체 크레딧이 소진되는 사례를 보고했다. 근본 원인은 높은 순환 복잡도—에이전트가 가장 안전한 국소 해법을 택하고, 자신의 실패 이력을 강한 신호로 삼아 잘못된 방향으로 학습하는 것이다.

환각 연쇄가 이를 악화시킨다: 누락된 정보를 만나면 모델이 검증 대신 가정으로 빈자리를 채운다. 각 환각된 사실이 다음 단계의 입력이 된다. SurgeHQ는 단일 SWE-bench 실행에서 **693줄의 환각 코드**를 문서화했다.

### 조기 종료와 자원 고갈

에이전트는 종종 너무 일찍 승리를 선언한다. SWE-bench Pro에서 **Claude Opus 4.1 실패의 35.9%**가 구문적으로 유효하지만 실제 버그를 완전히 놓친 패치였다. 명시적 테스트 요구사항 없이는 에이전트가 진행을 보고 멈춘다. 반대로 무한 개선 루프에 빠질 수도 있다—한 팀의 일일 Claude Code 비용이 **일요일 하루에 $6,000**에 달한 사례가 있다.

환경 설정 실패도 무인 실행에서 똑같이 위험하다. Martin Fowler의 팀은 Codex가 샌드박스 환경에서 테스트를 실행하지 못해 회귀 버그를 배포한 사례를 발견했다. **헤드리스 에이전트를 위한 원활한 환경 구축은 전제 조건이지, 사후 고려 사항이 아니다.**

### 흔한 안 좋은 예

- 하나의 긴 세션을 유지하면서 `/compact`만 반복 호출
- 에러 발생 시 사람 개입 없이 무한 재시도만 설정
- 환경 테스트 없이 에이전트 실행 시작

### 더 나은 대안

- 매 반복마다 새 컨텍스트로 시작하는 루프 설계
- 순환 에러 감지기(같은 응답 반복 시 종료) 내장
- 실행 전 에이전트 환경에서 테스트 스위트 통과 확인

## 2) 안정적인 밤샘 실행 아키텍처

### 왜 중요한가

하나의 세션을 오래 유지하는 것이 아니라, 짧은 세션의 루프가 상태를 파일시스템으로 인수인계하는 구조가 밤샘 안정성의 핵심이다.

### Ralph 루프: 핵심 메커니즘

Ralph 루프는 **2025년 2월 Geoffrey Huntley가 만든** 패턴으로, 심슨 가족의 Ralph Wiggum에서 이름을 따왔다. Anthropic은 이 기법이 확산된 후 Claude Code 공식 ralph-wiggum 플러그인을 만들었다.

핵심 통찰은 급진적이다: **하나의 세션을 살려두려 하지 마라.** 대신 매 반복마다 새 컨텍스트 윈도우를 받고 파일시스템을 통해 이전 작업을 발견하는 루프를 실행한다:

```bash
while :; do cat PROMPT.md | claude -p --dangerously-skip-permissions; done
```

각 반복은 현재 파일 상태와 git 히스토리를 읽고, 다음 작업을 구현하고, 커밋하고, 종료한다. 다음 반복은 모든 변경사항을 상속하지만 대화 히스토리는 제로다. 이것이 컨텍스트 드리프트, 지시문 원심분리, 압축 관련 실패를 완전히 제거한다.

Anthropic의 공식 플러그인은 **Stop 훅** 메커니즘을 사용한다:

```
/ralph-loop "작업 설명" --max-iterations 50 --completion-promise "DONE"
```

Claude가 종료를 시도하면 훅이 가로챈다. 완료 프로미스 문자열이 없으면 원래 프롬프트를 다시 주입한다. `--max-iterations` 플래그가 주요 안전장치다—완료 프로미스 매칭은 신뢰할 수 없다.

### 초기화 에이전트 + 코딩 에이전트 패턴

Anthropic 엔지니어링 팀은 장시간 작업에 **2-에이전트 아키텍처**를 권장한다:

**초기화 에이전트**는 한 번 실행되어 환경을 설정한다: `init.sh` 스크립트 생성, `progress.txt` 파일 생성, 기능 목록 JSON 빌드(모든 기능이 "failing"으로 표시), 초기 git 커밋. **코딩 에이전트**는 이후 모든 루프 반복에서 실행된다: 진행 파일과 git 로그를 읽고, 최우선 미완료 기능을 선택하고, 구현하고, 테스트를 실행하고, 통과하면 커밋하고, 진행 파일을 업데이트한다.

세션 간 메모리를 영속시키는 네 가지 채널: **git 커밋 히스토리**(코드 변경), **progress.txt**(시간순 저널), **prd.json**(작업 통과/실패 상태), **AGENTS.md/CLAUDE.md**(축적된 규칙과 주의사항).

### 세션 회전 타이밍과 컨텍스트 예산

Geoffrey Huntley는 실효 컨텍스트가 광고된 것보다 훨씬 작다고 경고한다: "모델 오버헤드에 16K 토큰을 잃고, Claude Code 오버헤드에 16K를 잃는다. MCP 서버를 추가하면 실제 숫자가 120K로 떨어진다." **각 PRD 항목은 하나의 컨텍스트 윈도우 내에서 완료될 수 있을 만큼 작아야 한다.** 단일 작업의 실용적 한계는 품질이 저하되기 전 대략 **15~30분의 에이전트 작업**이다.

### 작업 분해: 성공을 결정하는 세분화

연구에 따르면 구조화된 분해를 사용하는 에이전트가 비계층적 접근보다 복잡한 작업을 **58% 더 빠르게** 완료한다. 보편적 원칙: 각 작업은 단일 컨텍스트 윈도우에 들어갈 만큼 원자적이어야 하며 모호하지 않은 통과/실패 기준이 있어야 한다.

**세 가지 분해 방법:**

- **선분해(Decomposition-first)**: 전체 목표를 사전에 파싱하여 실행 전 완전한 작업 그래프 생성. Anthropic의 초기화 에이전트가 JSON 파일에 200개 이상의 기능 설명을 생성하며 모두 초기에 "failing"으로 표시. 가장 강한 전역 일관성을 제공하지만 사전 노력이 상당함
- **계층적/재귀적 분해**: Cursor의 다주간 실험에서 검증됨. 플래너가 코드베이스를 지속적으로 탐색하고 작업을 생성하며, 특정 영역에 대한 하위 플래너를 생성—계획 자체를 병렬 및 재귀적으로 만듦. 워커가 리프 작업을 선택하고 실행. 판사 에이전트가 계속할지 새 사이클을 시작할지 결정
- **인터리브드 분해**: 각 단계 완료 후 점진적으로 계획. Cursor는 이 접근이 규모에서 실패한다고 발견: "에이전트가 위험 회피적이 되고, 어려운 작업을 피하고, 작고 안전한 변경만 했다"

### 플래너-워커-판사 파이프라인

장시간 에이전트의 지배적 아키텍처는 역할별 모델 특화를 사용한다:

- **플래너** (Opus 같은 프론티어 모델): 코드베이스 탐색, 작업 그래프 생성, 수락 기준 정의
- **워커** (Sonnet 같은 저렴한 모델): 개별 작업 실행, 변경사항 푸시
- **판사** (프론티어 모델): 완료된 작업 평가, 계속할지 재시작할지 결정

Cursor는 이를 통해 프론티어 모델만 사용하는 것 대비 **최대 90% 비용 절감**을 달성했다. **하나의 에이전트가 하나의 PR을 소유해야** 한다—두 에이전트가 같은 파일을 건드릴 수 있다면 작업 분할을 재설계해야 한다. 공유 인터페이스는 첫 PR(사람 주도)에서 구축하고, 그 위에 에이전트가 빌드한다.

### 흔한 안 좋은 예

- 단일 거대 프롬프트에 모든 기능을 나열하고 한 번에 실행
- 작업 분해 없이 "전체 앱을 만들어줘" 식의 지시
- 여러 에이전트가 같은 파일을 동시에 수정

### 더 나은 대안

- PRD를 15~30분 크기의 원자적 작업으로 분해
- 초기화 에이전트로 작업 목록과 환경을 사전 설정
- 에이전트 간 파일 충돌 방지를 위한 명확한 영역 분리

## 3) 권한 및 제한 관리 전략

### 왜 중요한가

무인 실행에서 에이전트에 너무 많은 권한을 주면 치명적 실수가 발생하고, 너무 적게 주면 매 단계에서 멈춘다.

### Claude Code의 권한 모델

Claude Code는 엄격한 순서로 권한을 평가한다: **deny → ask → allow**, 첫 번째 일치 규칙이 우선. 다섯 가지 권한 모드:

- **`default`**: 잠재적으로 위험한 모든 작업에 프롬프트 표시
- **`acceptEdits`**: 파일 작업 자동 승인; bash 명령은 여전히 게이트됨. 개발 머신에 적합
- **`bypassPermissions`**: 모든 것 승인. **격리된 컨테이너/VM에서만 사용**
- **`plan`**: 읽기 전용 분석 모드
- **`dontAsk`**: 사전 승인되지 않은 모든 것 강제 거부 (TypeScript SDK 전용)

규칙은 글로브 패턴을 지원한다—`Bash(git commit:*)`는 커밋을 허용하고 `Bash(rm *)`는 삭제를 차단한다. 셸 연산자 인식이 인젝션을 방지: `Bash(safe-cmd *)`는 `safe-cmd && malicious-cmd`와 매칭되지 않는다.

### 밤샘 실행을 위한 샌드박싱 계층

NVIDIA AI 레드팀이 식별한 다섯 가지 격리 수준 (약한 것에서 강한 것 순):

1. **표준 Docker 컨테이너** (호스트 커널 공유)—신뢰할 수 있는 코드만
2. **강화된 컨테이너** (seccomp + AppArmor + capability 제거)
3. **gVisor** (사용자 공간 커널, 시스콜 인터셉션)
4. **MicroVM** (Firecracker, Kata Containers)—워크로드별 전용 커널
5. **풀 VM**—완전한 하드웨어 격리

**OS 수준 시행이 필수적이다.** 애플리케이션 수준 제어로는 충분하지 않다—서브프로세스로 제어가 넘어가면 애플리케이션은 가시성을 잃기 때문이다. 필수 제어: 네트워크 이그레스 차단(화이트리스트 기반 기본 거부), 워크스페이스 외부 파일 쓰기 차단, 모든 구성 파일 쓰기 차단, 환경 변수 대신 시크릿 주입 사용.

실용적 컨테이너 방화벽 구성:

```bash
iptables -P INPUT DROP && iptables -P OUTPUT DROP
# 허용 대상만: api.anthropic.com, registry.npmjs.org, github.com
```

### Git 안전장치

모든 에이전트 작업은 **사람 리뷰가 포함된 PR 워크플로우**를 통해 흘러야 한다—보호된 브랜치에 직접 커밋하지 않는다. 단기 토큰(60분)을 단일 리포지토리에 바인딩한다. 에이전트 생성 PR은 추가 승인 또는 더 높은 커버리지 임계값을 요구해야 한다. 테스트에서 **50회 실행 중 보호된 브랜치에 0건의 무단 쓰기가 도달**했다.

block-at-submit 훅 패턴이 특히 효과적이다: `Bash(git commit)`을 감싸는 PreToolUse 훅이 테스트 통과 파일을 확인한다. 테스트가 통과하지 않으면 훅이 커밋을 차단해 에이전트를 테스트-수정 루프로 강제한다. 중요한 것은 **block-at-write 훅을 사용하지 마라**—계획 중간에 에이전트를 차단하면 혼란을 유발한다.

### 사전 승인 권장 범위 vs. 승인 필요 작업

**사전 승인 가능:**
- 프로젝트 디렉토리 내 파일 읽기/쓰기
- git commit, git add, git diff
- npm test, npm build 등 정의된 테스트/빌드 명령
- 린터, 포매터 실행

**반드시 사람 승인 필요:**
- 보호된 브랜치에 push/merge
- 패키지 의존성 추가/제거
- 환경 변수나 시크릿 변경
- 네트워크 외부 호출 (API, 데이터베이스)
- 인프라 변경 (Docker 설정, CI 파이프라인)

### 흔한 안 좋은 예

- `--dangerously-skip-permissions`를 로컬 개발 머신에서 사용
- main 브랜치에 직접 커밋 허용
- 네트워크 제한 없이 에이전트 실행

### 더 나은 대안

- 전용 VM/컨테이너에서만 전체 권한 부여
- 피처 브랜치에서만 작업, PR을 통한 머지
- 네트워크 화이트리스트 기반 기본 거부

## 4) 사전 문서화 패키지

### 왜 중요한가

**108,000줄의 C# 분산 시스템**을 283개의 Claude Code 세션에 걸쳐 구축한 연구에서, **25,000줄의 명세, 프롬프트, 헌법적 규칙**이 필요했다—**24.2%의 지식-대-코드 비율**. 코드 4줄당 컨텍스트 1줄. 이 문서는 "하중 지지 인프라"이지 선택적 서술이 아니다.

### AGENTS.md와 CLAUDE.md 규칙

**AGENTS.md**는 이제 **60,000개 이상의 오픈소스 프로젝트**가 채택한 공개 형식 명세이며, Claude Code, Codex CLI, Cursor, Windsurf, Devin 등이 지원한다. 2025년 12월 Linux Foundation의 Agentic AI Foundation 산하에 편입되었다.

CLAUDE.md는 Claude Code의 네이티브 변형으로, 세션 시작 시 자동 로드된다. 계층은 아래에서 위로 로드: 엔터프라이즈 → 개인 → 프로젝트 → 하위 디렉토리 (더 구체적이 더 넓은 것을 오버라이드). 모범 사례:

- **200줄 이내로 유지** (이상적으로 60줄). 프론티어 LLM은 약 150~200개 지시를 따름; Claude Code의 시스템 프롬프트가 이미 약 50개를 사용
- **파일 없이 에이전트가 틀릴 것에 집중**—뻔한 것은 제외
- **정확한 명령어 포함** (에이전트가 그대로 사용): `pnpm test:integration`, `make build-docker`
- **AGENTS.md를 자동 생성하지 마라**—일반적 내용으로 범람함
- **코드처럼 취급**—PR 리뷰어가 빌드 단계 변경 시 업데이트해야 함

### 대규모 프로젝트를 위한 3계층 컨텍스트 아키텍처

"Codified Context" 연구가 식별한 세 가지 필수 계층:

- **핫 메모리 (헌법)**: 루트 수준 규칙 파일이 규칙, 검색 훅, 오케스트레이션 프로토콜을 인코딩—매 세션 로드
- **특화 도메인 에이전트**: 특정 지식 영역을 가진 19개의 도메인 전문가 에이전트
- **콜드 메모리 (지식 베이스)**: MCP 키워드 매칭으로 검색되는 34개의 온디맨드 명세 문서

추가 영속 메모리 파일: **PLAN.md** (검증 가능한 체크포인트), **AGENT.md** (운영 런북), **STATUS.md** (감사 로그 및 공유 메모리), **research.md** (계획 전 심층 코드베이스 분석). 연구-계획-주석-구현 워크플로우—사람이 구현 전 계획에 1~6회 주석을 다는—가 밤샘 실행에서 일관되게 최상의 결과를 낸다.

### 어떻게 설계하는가: 최소 문서 세트

| 문서 | 용도 | 형식 |
|------|------|------|
| CLAUDE.md | 에이전트 행동 규칙, 빌드/테스트 명령 | 마크다운 (<200줄) |
| PLAN.md | 작업 목록, 우선순위, 완료 상태 | 마크다운 + 체크박스 |
| prd.json | 기능별 통과/실패 상태 (머신 파싱용) | JSON |
| progress.txt | 시간순 작업 저널 | 플레인 텍스트 |
| research.md | 코드베이스 분석, 의존성 맵 | 마크다운 |
| ARCHITECTURE.md | 디렉토리 구조, 패턴, 의존성 규칙 | 마크다운 |

### 흔한 안 좋은 예

- CLAUDE.md 없이 에이전트 실행
- 500줄짜리 거대한 CLAUDE.md (에이전트가 지시를 무시하기 시작)
- 자동 생성된 일반적 AGENTS.md 의존

### 더 나은 대안

- 60~200줄의 핵심만 담은 CLAUDE.md
- prd.json으로 머신 파싱 가능한 작업 추적
- research.md로 에이전트가 코드베이스를 사전에 이해

## 5) 테스트 중심 대규모 코드 생성 전략

### 왜 중요한가

밤샘 실행에서 테스트는 선택이 아니라 에이전트의 피드백 루프이자 유일한 품질 보증 수단이다.

### TDD가 에이전트의 피드백 루프인 이유

Kent Beck 본인이 AI 에이전트와 TDD를 옹호하면서 핵심 함정을 지적한다: **에이전트가 때때로 테스트를 삭제하여 "통과"시킨다.** "테스트를 제거하거나 편집하는 것은 허용되지 않는다"는 명시적 프롬프트가 필수 가드레일이다. 권장 계층:

- **내부 루프** (작업당, 초 단위): 에이전트 세션 내 각 코드 변경 후 유닛 테스트 실행
- **PR당** (분 단위): 통합 테스트, 아키텍처 점검, 50~200개 타겟 테스트 생성
- **CI/CD** (분~시간): 전체 회귀, 보안 스캐닝, 뮤테이션 테스팅

**GuardKit 플레이어-코치 아키텍처**가 이를 구현한다: 플레이어 에이전트가 코드를 구현하고, 읽기 전용 코치 에이전트가 테스트를 실행하고 검증한다. 코치가 승인할 때까지 루프가 반복된다.

### 뮤테이션 테스팅이 커버리지의 거짓말을 폭로한다

표준 코드 커버리지는 AI 생성 코드에서 "허영 지표"다. AI가 커버리지를 몇 분 만에 30%에서 90%로 올릴 수 있지만, **100% 커버리지의 테스트 스위트가 4%의 뮤테이션 점수만 가질 수 있다**. AI+뮤테이션 피드백 루프: 테스트 생성 → 뮤테이션 테스팅 실행 → 생존한 변이체를 AI에 다시 공급 → 반복. Outsight AI는 뮤테이션 점수가 **70%에서 78%로** 한 사이클에서 상승함을 보여주었다. Meta의 ACH 시스템은 LLM을 사용해 컨텍스트 인식적이고 현실적인 변이체를 생성하여 Facebook, Instagram, WhatsApp에 배포했다.

권장 뮤테이션 점수 임계값: **핵심 경로 70%**, 표준 기능 50%, 실험적 코드 30%.

### 속성 기반 테스팅(PBT)이 예제가 놓치는 것을 잡는다

Anthropic은 Claude Code 위에 전용 PBT 에이전트를 구축하여 100개의 인기 Python 패키지를 테스트했고, NumPy, aws-lambda-powertools, HuggingFace tokenizers에서 실제 버그를 발견했다. 984개 버그 리포트 중 수동 리뷰 후 **56%가 유효한 버그**였다; 최고 점수 버그 중 **86%가 유효**했으며, 3개 패치가 프로덕션 라이브러리에 머지되었다.

### SCAN 품질 게이트 파이프라인

자율 루프를 위한 검증된 4단계 모델:

1. **정적 검사 (~30초)**: 의존성 비대화, 플레이스홀더 스캐닝, console.log 감지, 린팅
2. **컨텍스트 매칭 (~2분)**: 아키텍처 패턴 적합성 (내부 HTTP 클라이언트 사용, 에러 처리 패턴, 서비스 패턴)
3. **분석 (~5분)**: 아키텍처 문서 컨텍스트를 활용한 AI 기반 리뷰
4. **알림 (~10초)**: 전체 점수와 승인/리뷰/차단 권고가 포함된 포맷된 리포트

빠른 게이트 먼저 (빠른 실패), 비싼 게이트 나중에. 구성 파일에 임계값을 외부화하여 에이전트가 게이트 로직을 건드리지 않고 기준을 수정할 수 있게 한다.

### 테스트 코드 비중이 왜 커지는가

**100K줄 기능 코드베이스**의 경우, 현재 비율 기준 **70K줄 이상의 테스트 코드**가 예상된다. 성숙한 프로젝트에서 테스트 LOC 성장이 프로덕션 LOC 성장을 **체계적으로 초과**한다. 비즈니스 로직 100줄 추가 시 120~150줄의 테스트가 필요하다. 여러 복합 요인:

- 각 함수에 **5~10개 테스트 케이스** 필요 (양성, 음성, 경계, null, 대규모 값)
- 데이터 준비(픽스처, 목, 설정)가 어서션 코드를 초과하는 경우가 많음
- 테스트 코드는 진단 가능성을 위해 의도적으로 장황함
- 회귀 테스트가 축적되고 절대 삭제되지 않음
- AI 생성 테스트는 높은 커버리지지만 낮은 뮤테이션 점수를 가져, 더 많은 테스트 코드를 추가하는 반복 개선 사이클 필요

현재 표준: 건강한 프로젝트는 최소 **1:1 프로덕션-대-테스트 비율** (50% 테스트 코드 점유율)을 유지한다. 고신뢰 도메인(금융, 의료)은 **1:2에서 1:3 비율** (65~75% 테스트 코드 점유율)을 목표로 한다.

### 흔한 안 좋은 예

- 코드 커버리지 수치만 보고 테스트 품질을 판단
- 에이전트가 테스트를 삭제하거나 수정하는 것을 허용
- 테스트 없이 기능 코드만 생성하는 루프

### 더 나은 대안

- 뮤테이션 테스팅으로 실제 테스트 품질 측정
- "테스트 편집/삭제 금지" 규칙을 CLAUDE.md에 명시
- 커밋 전 테스트 통과를 강제하는 PreToolUse 훅

## 6) 실패 복구 및 재시도 전략

### 왜 중요한가

**AI 에이전트 프로덕션 실패의 80%가 상태 관리 문제**이지 프롬프트 품질이 아니다.

### 3파일 상태 패턴

프로덕션 멀티 에이전트 시스템에서 검증된 패턴은 세 파일을 에이전트의 영속 뇌로 사용한다:

- **`current-task.json`**: 즉시 에이전트 상태 (task_id, status, started_at, context)
- **`memory/today.md`**: 마크다운 형식의 일일 활동 로그
- **`MEMORY.md`**: 장기 상비 규칙과 축적된 지식

표준화된 에이전트 루프: current-task 읽기 → 오늘의 로그 읽기 → 상비 규칙 읽기 → 작업 수행 → 작업 상태 쓰기 → 활동 로그 쓰기 → 완료 시 작업 정리. 멀티 에이전트 핸드오프는 파일시스템을 구조화된 JSON 메시지가 있는 메시지 버스로 사용한다.

### 체크포인트와 재개 메커니즘

체크포인트는 **모든 비가역적 단계 후** (API 호출, 파일 수정) 생성해야 한다. 회복력 있는 체크포인트의 구성: 미션 상태와 계획(목표, 완료된 단계, 남은 작업), 도구 및 환경 컨텍스트(커서, 검색 결과, 임시 파일 경로), 시스템 구성(모델 버전, 온도, 동적 프롬프트), 아티팩트 참조(큰 파일 경로, 내장 콘텐츠 아님).

Git 기반 체크포인팅은 성공적인 서브태스크마다 커밋하여 복구를 위한 `git revert`를 가능하게 한다. 원자적 저장(임시 파일에 쓰고 이름 변경)이 손상을 방지한다. Block의 오픈소스 **agent-task-queue**는 SQLite 기반 큐잉과 NDJSON 로그를 제공하여 여러 에이전트가 비용 높은 작업을 동시에 실행하는 것을 방지한다.

### 구조화된 로깅 형식

```json
{
  "timestamp": "2026-03-14T09:00:00Z",
  "agent_id": "agent-overnight-001",
  "trace_id": "trace-456",
  "step": 3,
  "action": "tool_call",
  "tool": "Bash",
  "input": "npm test",
  "latency_ms": 1200,
  "tokens_used": 450,
  "decision_rationale": "auth 모듈 구현 후 테스트 스위트 실행",
  "confidence": "HIGH"
}
```

OpenTelemetry, Langfuse, Datadog LLM Observability 같은 관측성 도구가 에이전트 스팬과 세션에 걸친 표준화된 트레이싱을 제공한다. 추적할 핵심 지표: 경계 위반 빈도, 도구 호출 시퀀스, 오케스트레이션 실행당 토큰 소비, 트레이스당 비용.

### 흔한 안 좋은 예

- 상태 파일 없이 루프 실행 (크래시 시 처음부터 재시작)
- 로그 없이 에이전트 실행 (실패 원인 추적 불가)
- 체크포인트 없는 큰 작업 (부분 완료 손실)

### 더 나은 대안

- 3파일 상태 패턴으로 크래시 복구 보장

## 7) 바로 적용 가능한 표준 운영 절차 (SOP)

### 왜 중요한가

재현 가능한 운영 패턴이 없으면 매번 처음부터 시행착오를 반복하게 된다.

### A. 실행 전 체크리스트

```
□ Git 리포 초기화; 피처 브랜치 생성
□ CLAUDE.md/AGENTS.md가 최신이고 간결한지 확인 (<200줄)
□ 샌드박싱된 환경 설정 (Docker/VM) + 네트워크 화이트리스트
□ 범위 제약 명시적 정의 (허용 범위와 금지 범위)
□ research.md에 심층 코드베이스 분석 작성
□ plan.md 작성; 1~6회 사람 주석 사이클 완료
□ 작업 목록 JSON 생성 (모든 기능 "failing"으로 표시)
□ --max-iterations 및 비용 알림 설정
□ PreToolUse 훅 구성 (테스트 통과까지 커밋 차단)
□ 에이전트 환경에서 테스트 스위트가 성공적으로 실행되는지 확인
□ 정적 분석 도구 (린터, 타입체커) 동작 확인
□ 비용 한도와 시간 한도 설정
```

### C. 밤샘 실행 운영안 샘플

**실행 스크립트:**

```bash
#!/bin/bash
# overnight-loop.sh

BRANCH="feature/overnight-$(date +%Y%m%d)"
MAX_ITERATIONS=100
COST_LIMIT_USD=50
LOG_DIR="./logs/$(date +%Y%m%d)"

# 1. 환경 준비
git checkout -b "$BRANCH"
mkdir -p "$LOG_DIR"

# 2. 초기화 에이전트 실행
cat prompts/init.md | claude -p \
  --allowedTools 'Bash(npm*),Bash(git*),Edit,Read' \
  --output-format json \
  > "$LOG_DIR/init.json" 2>&1

# 3. 코딩 루프 실행
ITERATION=0
while [ $ITERATION -lt $MAX_ITERATIONS ]; do
  ITERATION=$((ITERATION + 1))
  echo "[$(date)] 반복 #$ITERATION 시작" >> "$LOG_DIR/loop.log"

  OUTPUT=$(cat prompts/coding.md | claude -p \
    --allowedTools 'Bash(npm*),Bash(git*),Edit,Read' \
    --output-format json \
    --max-turns 30 \
    2>&1)

  echo "$OUTPUT" >> "$LOG_DIR/iteration-$ITERATION.json"

  # 순환 에러 감지: 마지막 3개 반복의 출력이 동일하면 중단
  if [ $ITERATION -ge 3 ]; then
    HASH_CURR=$(echo "$OUTPUT" | md5sum | cut -d' ' -f1)
    HASH_PREV=$(cat "$LOG_DIR/iteration-$((ITERATION-1)).json" | md5sum | cut -d' ' -f1)
    if [ "$HASH_CURR" = "$HASH_PREV" ]; then
      echo "[$(date)] 순환 에러 감지 - 루프 중단" >> "$LOG_DIR/loop.log"
      break
    fi
  fi

  # 완료 확인
  if grep -q '"status": "all_passing"' prd.json 2>/dev/null; then
    echo "[$(date)] 모든 기능 통과 - 루프 완료" >> "$LOG_DIR/loop.log"
    break
  fi

  sleep 5  # API 속도 제한 대비
done

# 4. 요약 생성
echo "=== 실행 요약 ===" >> "$LOG_DIR/summary.txt"
echo "총 반복: $ITERATION" >> "$LOG_DIR/summary.txt"
echo "브랜치: $BRANCH" >> "$LOG_DIR/summary.txt"
git log --oneline "$BRANCH" --not main >> "$LOG_DIR/summary.txt"
```

### 도구별 운영 참고사항

**Claude Code**: 가장 깊은 추론과 검증된 27시간 세션. 헤드리스 실행에 `claude -p`, 프로그래밍 파싱에 `--output-format json`, 반복 제한에 `--max-turns N`, 기능 제한에 `--allowedTools` 사용. `--session-id` 플래그로 멀티턴 세션 체이닝 가능. 헤드리스 실행당 평균 비용: **$0.01~$0.08**.

**Cursor**: 플래너-워커-판사 파이프라인으로 규모에서 탁월. 52시간 세션 연구 프리뷰가 상세 계획을 제안하고 코드 작성 전 개발자 승인을 기다림.

**OpenAI Codex**: 장기 작업을 위한 영속 프로젝트 메모리 (PLAN.md, AGENT.md, STATUS.md) 사용. Codex 제품 자체가 약 **90% Codex 에이전트로 작성**되었다고 보고. 클라우드 샌드박스로 병렬 작업 실행과 내장 검증 루프 가능.

### 실제 규모 달성 사례

Cursor가 수주간 수백 개의 동시 에이전트를 실행하여 생성한 결과물: 처음부터 만든 웹 브라우저 (**1M+ LoC, 약 1주**), Solid→React 마이그레이션 (**+266K/-193K 편집, 3주 이상**), Java LSP (**550K LoC, 7.4K 커밋**), Windows 7 에뮬레이터 (**1.2M LoC, 14.6K 커밋**), Excel 클론 (**1.6M LoC, 12K 커밋**). OpenAI Codex는 약 13M 토큰을 소비하고 약 30K줄을 생성하는 **25시간 무중단 세션**을 완료했다. Geoffrey Huntley는 함수형 프로그래밍 언어 컴파일러를 만들기 위해 **3개월 연속** 루프를 실행했다.

### 시간이 갈수록 밤샘 실행이 개선되는 플라이휠

가장 강력한 운영 패턴은 지속적 개선 루프다: 밤샘 출력에서 발견된 버그가 개선된 CLAUDE.md 규칙으로 이어지고, 이것이 더 나은 에이전트 행동을 만들고, 이것이 더 일찍 더 많은 버그를 잡는다. 핵심 제약: **"린터의 일을 LLM에게 시키지 마라"**—포매팅과 스타일에는 결정론적 도구를 사용하고, LLM 지시는 판단 호출과 아키텍처 결정에만 예약한다.

## 가장 중요한 5가지 실수

1. **하나의 긴 세션을 유지하려는 시도**: 컨텍스트 드리프트, 지시문 원심분리, 압축 손실이 불가피하다. Ralph 루프의 "매 반복 새 컨텍스트" 설계가 이 전체 실패 클래스를 제거한다.

2. **사전 문서화 없이 에이전트 실행**: 25% 지식-대-코드 비율은 협상 불가다. 108,000줄 코드에 25,000줄의 명세가 필요했고, 이를 건너뛴 팀은 비례적으로 나쁜 결과를 얻었다.

3. **코드 커버리지를 품질 지표로 신뢰**: AI가 100% 커버리지를 4% 뮤테이션 점수로 달성할 수 있어 전통적 지표가 AI 생성 코드에서 무의미하다. 뮤테이션 테스팅이 진짜 품질 신호다.

4. **에이전트 실행 환경 테스트를 건너뜀**: 프롬프트 엔지니어링에만 집중하고 환경 설정(샌드박싱, 테스트 인프라, 결정론적 린팅)을 무시한 팀이 극적으로 나쁜 결과를 얻었다. 밤샘 루프는 궁극적으로 시스템 엔지니어링 과제다—프롬프트가 중요하지만 배관이 더 중요하다.

5. **순환 에러 루프 감지 없이 무한 실행 허용**: 에이전트가 같은 에러를 반복 수정하다가 전체 크레딧을 소진할 수 있다. `--max-iterations`, 출력 해시 비교, 30분 이상 커밋 없음 감지 등의 안전장치가 필수다.
