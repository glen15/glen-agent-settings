CC의 공식 플러그인 ralph-loop과 기존 창안자가 하는 방식의 차이와 장단점
CC 플러그인: 세션 내에서 작동. Stop Hook을 가로채서 프롬프트 재실행. 어쩔 수 없겠죠? 세션 안이니까. 이 방법뿐. 절반 이상은 갑니다. 왜냐면 ralph의 핵심인 '이전 루프에서의 실수를 기록하고, 현재 상태(progress)를 기록'해야 하는 일을 CC가 대신 해줍니다. 다만 컨텍스트가 계속 누적됨. auto-compact가 실행되지만 본질적으로 손실압축임. 컨텍스트 오염을 피하고자 했던 걸 막을 수는 없음. 비교적 가벼운 작업이나, 초기 구현에는 쓸만함. 
오리지널 ralph: bash나  python 등의 언어로 loop 제어. 매 컨텍스트가 fresh하게 진행. 대신 memory(progress, state 등)를 직접 관리해야 합니다. 그래서 보통 2-단계 아키텍처를 권장합니다. 첫번째 단계는 init을 하는거죠. prd, progess, state 등을 만듭니다. 그리고 2번째가 메인 루프를 수행하는 것이죠. 이 패턴이 필요한 이유는, 에이전트가 "무엇을 해야 하는지" 스스로 파악해야 하기 때문! (코드 생성 작업에는 필수입니다)

API가 아니고 구독 요금제에서 밤샘 작업할 건데, usage limit을 어떻게 관리하지?
돈이 썩어나면 그냥 extra usage 켜고 잠들면 되는데, 너무 아깝습니다. 
다들 아시듯, 구독 요금제 하에서는 rolling window가 적용돼서 대략 5~6시간 마다 리밋을 관리하죠. 밤새 작업하다가 이 리밋 만나면 우짜지?
매우 간단합니다. 그냥 exponential back-off 하면 됩니다. 
usage limit, exceed 같은 키워드 필터 걸어서 30분 sleep, 하고 또 하면 30분 더 sleep, 또 하면 60분 sleep 이런식입니다.
