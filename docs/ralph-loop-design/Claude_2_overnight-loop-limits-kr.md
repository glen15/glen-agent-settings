# 밤샘 Claude Code 루프: 리밋, 플러그인, 그리고 실제로 작동하는 것

**Claude Max $200 플랜은 주당 Sonnet 약 240~480시간, Opus 약 24~40시간을 제공하지만, Claude Code는 매 에이전트 턴마다 전체 대화 컨텍스트를 재전송하기 때문에 웹 채팅보다 5~100배 빠르게 리밋을 소모한다.** 밤샘 자율 루프에서는 결정적인 아키텍처 선택에 직면한다: Ralph Wiggum 플러그인은 Claude를 단일 세션에 유지하면서 컨텍스트가 누적되고("컨텍스트 부식" 위험), bash 래퍼 루프(`while :; do cat PROMPT.md | claude -p; done`)는 매 반복마다 깨끗한 200K 토큰 윈도우를 제공하는 대신 대화 메모리를 잃는다. 실제 운영 경험은 밤샘 안정성을 위해 신선한 세션 bash 루프를 압도적으로 지지하지만, 두 접근법 모두 Max 플랜에서 신중한 관리 없이는 레이트 리밋에 부딪힌다.

## 질문 1: $200 Max 플랜의 이중 레이어 레이트 리밋 시스템

Claude Max 20x ($200/월)는 **Pro 플랜 대비 20배의 사용량**을 claude.ai, Claude Code, Claude Desktop 간 공유되는 단일 풀로 제공한다. 별도의 Claude Code 할당량은 없다—모든 표면이 같은 버킷에서 뽑아 쓴다. Anthropic은 이를 동시에 작동하는 두 개의 독립적 리밋으로 구성한다.

**5시간 롤링 윈도우**는 개인용이며 첫 메시지를 보낼 때 시작된다. 이 윈도우 내에서 Max 20x 사용자는 대략 **200~800개의 Claude Code 프롬프트**를 받는다 (Pro에서는 약 10~40개). 타이머는 고정 스케줄이 아니라 첫 메시지 전송 5시간 후에 리셋된다. **주간 리밋**은 총 컴퓨트를 7일 롤링 기간 동안 약 **Sonnet 4 기준 240~480시간, Opus 4 기준 24~40시간**으로 제한한다. 두 리밋 모두 내부적으로 토큰 기반이다—"시간"은 Anthropic의 대략적 변환이다. Anthropic은 "Claude Code를 백그라운드에서 24/7 연속으로 돌리는" 사용자를 억제하기 위해 주간 리밋을 도입했다.

대부분의 개발자가 놓치는 핵심 디테일: **Claude Code는 웹 채팅과 극적으로 다른 속도로 토큰을 소비한다**. 단일 Claude Code 인터랙션이 10,000~50,000+ 토큰을 소모할 수 있는데, 에이전트 루프에 파일 읽기, 편집, bash 명령 실행, 전체 대화 히스토리 재생이 포함되기 때문이다. 한 분석에 따르면, 15개의 반복 명령을 내리는 개발자의 마지막 명령이 전체 대화 히스토리가 포함되어 **200,000+ 입력 토큰**을 보내는 경우가 있었다. Max 20x에서 두 프로젝트를 동시에 실행하면 레이트 리밋 윈도우를 1시간 이내에 소진할 수 있다.

### 세션 중간에 리밋에 도달하면 어떻게 되나

Claude Code 세션 중 레이트 리밋에 도달하면, **Claude Code가 새 프롬프트를 즉시 차단**한다—하지만 세션이 종료되지는 않는다. 대화 히스토리는 그대로 남고, `"Claude usage limit reached. Your limit will reset at [날짜/시간]."`이라는 메시지가 표시된다. 밤샘 자율 루프의 경우, 이것은 치명적 실패 모드를 만든다: bash 래퍼 루프는 계속 재시도하고 (Claude가 에러로 종료하면 루프가 재호출), Ralph Wiggum 플러그인의 stop 훅은 종료를 차단하려 하지만 Claude가 진행할 수 없다.

### 리밋 도달 시 대처법 3가지

**1) Extra Usage 활성화**: API 요율 과금으로 원활하게 오버플로. Sonnet 4.6은 입력/출력 백만 토큰당 $3/$15, Opus 4.6은 $5/$25. Claude Console에서 사전 충전과 명시적 사용자 동의가 필요하다. 밤샘 루프에서 리밋 걱정 없이 돌리려면 이것이 가장 확실한 보험이다.

**2) 리셋 대기**: 5시간 윈도우 또는 주간 리셋을 기다린다. 밤샘 루프 스크립트에 리밋 감지 시 sleep을 넣어 자동 대기 후 재시작하는 로직을 구현할 수 있다:

```bash
while :; do
  OUTPUT=$(cat PROMPT.md | claude -p --max-turns 30 2>&1)
  EXIT_CODE=$?

  # 레이트 리밋 감지 시 대기
  if echo "$OUTPUT" | grep -q "rate limit\|usage limit"; then
    echo "[$(date)] 레이트 리밋 도달 - 30분 대기 후 재시도" >> loop.log
    sleep 1800  # 30분 대기
    continue
  fi

  # 정상 종료 또는 완료 확인
  if grep -q '"status": "all_passing"' prd.json 2>/dev/null; then
    echo "[$(date)] 모든 기능 통과 - 루프 완료" >> loop.log
    break
  fi

  sleep 5
done
```

**3) 모델 전환 전략**: **계획(planning)에는 Opus를 아끼고, 구현(implementation)에는 Sonnet을 사용**한다. Sonnet의 리밋이 극적으로 더 높기 때문이다. 밤샘 루프에서는 Sonnet으로 대부분의 코딩 작업을 돌리고, Opus는 아키텍처 결정이나 복잡한 디버깅에만 예약한다.

### 리밋 모니터링 방법

`/usage` (5시간 및 주간 활용도와 리셋 시간 표시)와 `/status` (플랜 타입과 세션 퍼센트 표시)로 모니터링할 수 있다. 하지만 **리밋 접근 시 선제적 경고가 내장되어 있지 않다**—이것은 오픈 피처 리퀘스트다. 기저 API가 `five_hour.utilization`과 `seven_day.utilization`을 0~1 float로 노출하며, 터미널 상태줄에 이를 표시하는 커뮤니티 스크립트가 있다. 알려진 버그: 16%나 32% 사용량에서 잘못된 레이트 리밋 에러 발생 (분당 처리량 리밋과 세션 할당량 혼동), 잔여 플랜 할당량에도 불구하고 Extra Usage가 잘못 과금되는 케이스.

## 질문 2: Ralph Wiggum 플러그인은 세션 내부에서 동작하는가?

### 답: 그렇다. 세션 내부에서 동작하며, 이것이 핵심 차이점이다.

Ralph Wiggum 플러그인은 Claude Code의 Stop 훅 시스템을 사용하여 **완전히 단일 Claude Code 세션 내에서 동작**한다. 세션을 재시작하지 않는다. `/ralph-loop "작업 설명" --max-iterations 20 --completion-promise "DONE"`을 실행하면, 플러그인이 `.claude/ralph-loop.local.md`에 상태 파일을 생성한다. 이 파일에는 YAML 프론트매터(반복 횟수, 최대 반복, 완료 프로미스)와 프롬프트 본문이 담긴다.

핵심 메커니즘은 Claude가 작업이 끝났다고 판단하고 종료를 시도할 때 발동한다. 플러그인의 `stop-hook.sh` 스크립트가 이 종료를 가로채고, 상태 파일을 읽고, 두 가지 조건을 확인한다: 반복 리밋에 도달했는가, 또는 Claude가 `<promise>` 태그 안에 완료 프로미스 문자열을 출력했는가? 두 조건 모두 충족되지 않으면, 훅이 **반복 카운터를 증가시키고, stderr로 JSON 차단 결정을 출력하고, 종료 코드 2로 종료**한다—이것이 Claude Code에게 종료를 막고 원래 프롬프트를 Claude에게 연속 지시로 다시 공급하라고 알린다.

**반복 간에 컨텍스트가 누적된다**—같은 세션이기 때문이다. 이것이 근본적인 아키텍처적 구분이다. Claude의 "자기 참조"는 출력-입력 피드백이 아니라 파일시스템에서 온다: 각 반복이 같은 프롬프트를 보지만, Claude는 디스크에서 수정된 파일, git 히스토리, 자신의 이전 변경사항을 관찰한다. 대화 히스토리는 매 반복마다 커지고, 결국 자동 압축(auto-compaction)을 트리거한다.

### bash 래퍼 루프는 메모리를 안정성과 교환한다

bash 래퍼 접근법은 **대화 메모리가 없는 완전히 독립적인 세션**을 생성한다:

```bash
while :; do
  cat PROMPT.md | claude -p --max-turns 30
done
```

각 `claude -p` 호출은 **새로운 200K 토큰 컨텍스트 윈도우**로 시작한다. 반복 간 유일한 연속성은 파일시스템을 통해 영속된다: git 히스토리, 수정된 파일, 진행 추적 파일 (`prd.json`, `progress.txt`), CLAUDE.md (매 세션 자동 로드). 이것은 플러그인과 근본적으로 다르다—하나의 긴 세션에서 컨텍스트를 누적하는 대신, 각 반복이 전체 토큰 예산을 받는다.

`--max-turns` 플래그는 각 반복 내 도구 사용 라운드 트립을 제한한다. 리밋에 도달하면 Claude Code가 서브타입 `error_max_turns`인 `ResultMessage`를 반환하고 **종료 코드 1로 종료**한다 (0이 아님). `result` 필드는 채워지지 않는다—부분 작업은 Claude가 이미 디스크에 쓴 파일에만 존재한다.

Anthropic 자체 엔지니어링 가이드는 기능 추적에 Markdown보다 JSON을 권장한다—"모델이 Markdown에 비해 JSON 파일을 부적절하게 변경하거나 덮어쓸 가능성이 적다."

### 두 접근법의 트레이드오프 비교

| 차원 | Ralph Wiggum 플러그인 (세션 내부) | bash 래퍼 루프 (새 세션) |
|---|---|---|
| **반복당 컨텍스트** | 누적됨 (시간이 갈수록 품질 저하) | 매번 완전한 ~176K 사용 가능 토큰 |
| **반복 간 메모리** | 전체 대화 히스토리 | 파일, git, CLAUDE.md만 |
| **컨텍스트 저하** | O — 60%+ 활용 후 "컨텍스트 부식" | X — 매번 깨끗한 시작 |
| **압축 위험** | 높음 — 손실 요약이 추론 체인 삭제 | 없음 — 압축 필요 없음 |
| **실패 격리** | 혼란스러운 반복 하나가 모든 후속 반복을 오염 | 각 반복이 독립적 |
| **구현 복잡도** | 플러그인 설치 + 슬래시 커맨드 | 3줄 bash 스크립트 |
| **레이트 리밋 회복력** | 세션 차단됨; stop 훅이 강제 연속 불가 | 루프가 다음 반복에서 자동 재시도 |
| **적합 상황** | 누적 이해가 필요한 복잡한 상호연결 작업 | 많은 독립적 소규모 작업, 밤샘 안정성 |

단일 세션 접근법은 사용자들이 **"컨텍스트 부식"**이라 부르는 현상을 겪는다: 모든 것이 잘 되는 허니문 단계, 그 다음 저하 단계(응답 느려짐, 파일 재읽기, 압축 메시지), 그리고 회귀 단계(Claude가 최근 컨텍스트를 잊고, 완료된 코드를 다시 쓰고, 모순적 변경을 제안). 자동 압축은 손실 압축이다—50줄의 아키텍처 논의가 한 문장이 되고, 결정의 "이유"가 사라진다.

### 밤샘에서는 왜 bash 래퍼가 권장되는가

이유는 명확하다:

**첫째, 실패 격리.** 플러그인에서는 혼란스러운 반복 하나가 이후 모든 반복의 컨텍스트를 오염시킨다. bash 래퍼에서는 각 반복이 성공하거나 실패하거나—독립적이다. 다음 반복은 깨끗하게 시작한다.

**둘째, 컨텍스트 예산.** 밤샘 동안 수십~수백 반복을 돌린다면, 플러그인은 점점 줄어드는 유효 컨텍스트로 작업하게 된다. bash 래퍼는 매번 전체 176K 토큰을 쓴다.

**셋째, 레이트 리밋 회복.** 플러그인에서 레이트 리밋에 걸리면 세션이 멈추고 stop 훅이 아무것도 할 수 없다. bash 래퍼에서는 `claude -p`가 에러로 종료하면 루프가 sleep 후 다음 반복을 시도한다.

**넷째, 디버깅.** 플러그인에서 문제가 생기면 수백 턴의 대화 히스토리를 추적해야 한다. bash 래퍼에서는 각 반복의 출력이 독립적인 JSON 파일로 기록되어 정확히 어디서 문제가 생겼는지 추적이 쉽다.

### 108시간 무인 운영의 현실

한 개발자가 108시간 무인 운영을 문서화했는데, Claude가 `rm -rf ./src/`를 두 번 실행하고 ("깨끗한 상태로 정리"), 서브에이전트가 API 루프에 빠져 **1시간에 $8**를 소비하고, 에러 수정 루프가 20번 반복하면서 매번 "수정 완료"를 보고하고, 컨텍스트 윈도우 소진으로 Claude가 **3시간의 작업을 덮어쓰는** 사태가 발생했다. 또 다른 사용자는 자동 압축 재시도 루프가 2.5시간 동안 27개의 압축 서브에이전트 파일을 만들고 밤새 **4,246개의 API 요청**을 보낸 경험을 했다.

## 결론: 권장 밤샘 아키텍처

밤샘 루프에서의 증거는 명확한 모범 사례를 가리킨다: **Ralph Wiggum 플러그인이 아니라, 적극적인 가드레일을 갖춘 새 세션 bash 루프를 사용하라.** Anthropic 자체 엔지니어링 가이드가 2-에이전트 아키텍처를 권장한다: 초기화 에이전트가 `passes: false` 항목을 가진 포괄적 기능 목록을 JSON으로 생성하고, 코딩 에이전트가 세션당 하나의 기능에 대해 점진적 진행을 한다.

프로덕션 설정에 포함해야 할 것:

- **`--max-turns`를 반복당 안전망으로** (30이 일반적 선택)
- **별도 브랜치에서만 작업**, 테스트 실패 시 CI가 푸시를 차단
- **격리된 환경** (Docker 또는 일회용 VM)에서 `--dangerously-skip-permissions` 사용 시
- **Extra Usage를 오버플로 보험으로** 활성화
- **구현에 Sonnet, 아키텍처에 Opus** 예약
- **보수적 반복 리밋** 설정 (100이 아니라 10~20으로 시작)
- **레이트 리밋 감지 시 자동 대기** 로직을 bash 스크립트에 내장

약 **25%는 폐기**할 것으로 예상하라—네 번 중 한 번의 반복은 버릴 만한 출력을 생성한다. 머지 전 항상 리뷰하고, git을 타협 불가능한 안전망으로 유지하며, Claude에게 테스트 피드백 루프(테스트 스위트, 예상 출력)를 제공하는 것이 자율 품질을 위한 가장 레버리지 높은 투자다.
